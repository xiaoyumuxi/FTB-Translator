# CMP 写回事务

写回从用户确认应用 CMP 开始。这里的“事务”指应用层的预验证、备份、内存生成、提交快照和失败补偿；它不是由文件系统提供的跨文件原子事务。

## 安全目标

- API 请求和人工校对期间不修改任务书；
- 任何身份、源内容、位置或格式校验失败时，不创建备份、不写文件；
- 所有目标文件都能生成合法结果后才进入提交；
- 多文件提交中途失败时恢复本次提交前的内容；
- 已成功提交任务书后，历史或报告保存失败不能把成功误报为写回失败。

## 阶段 1：加载与身份校验

`cmp-apply` 先解析 CMP v1，并在状态库中把该任务从 `review_ready` 原子转换为 `applying`，再解析用户当前选择的任务书目录。`applying` 或 `applied` 状态会直接拒绝重复命令。应用要求 CMP 的规范化 `quests_dir` 和当前目录一致、`mode` 一致。早期 v1 没有 `task_id` 时使用稳定内部编号，否则沿用 CMP 编号。

此阶段仅读取文件。CMP 头、meta、状态、JSON 位置行、翻译对语法或重复 `(entry_id, path)` 无效时立即拒绝。

## 阶段 2：重新扫描源文件

应用不会相信生成 CMP 时留在内存里的扫描结果。它重新读取 `lang/en_us.snbt` 或排序后的 `chapters/*.snbt`，用与翻译阶段相同的 `prepare_entry` 规则重建条目和富文本单元，然后比较：

1. `total_entries`；
2. 由条目 ID、NUL 分隔符、完整英文和 `0xff` 分隔符计算的 SHA-256 `source_fingerprint`；
3. CMP 记录数与当前翻译单元数；
4. 每个 `(entry_id, path)` 是否存在且唯一；
5. `file` 是否由当前模式和条目 ID 推导得到；
6. 箭头左侧英文是否与当前单元原文完全相同。

目录、模式、任务书内容、文件归属、条目位置或英文被改动时，流程在任何备份之前终止。

## 阶段 3：译文与完整条目校验

每条右侧译文必须非空，并再次经过换行、格式 token 和富文本结构守卫。然后应用按 JSON Pointer 重建完整富文本条目，再对完整英文/译文运行同一守卫。记录状态不是豁免条件。

非 `translated` 状态仍保持英文时允许应用，但会进入最终报告的人工检查告警；这与“格式不安全仍强行写入”不同，英文原文本身仍须通过所有守卫。

## 阶段 4：在内存中生成全部输出

`lang` 模式复制当前有序 map，替换文本或按换行拆回字符串数组，生成完整 `zh_cn.snbt` 字符串，再用 `snbt::parse` 重新解析。目标只有 `lang/zh_cn.snbt`，源 `en_us.snbt` 不修改。

`chapters` 模式按源文件聚合 `(segment.index, target)`，要求位置无重复、全部存在；按字节 span 从后向前替换并正确转义原引号。每个章节在替换前后检查字符串和 `{}[]()` 结构。直到所有 `FileOutput` 都构造成功，磁盘仍未发生写回。

## 阶段 5：备份

所有验证通过后，应用才在 `.ftb-translator/backups/` 下原子争用一个未使用的目录名。基础名为本地时间 `YYYYMMDD-HHMMSS`，同秒冲突时追加 `-001` 至 `-999`，避免覆盖已有备份。

`lang` 模式复制整个当前 `lang/`，`chapters` 模式复制整个当前 `chapters/`。父目录创建、快照目录创建和逐文件复制分别返回结构化备份错误；复制中途失败会删除本次未完成的快照目录，然后在任何任务书写回前终止。

## 阶段 6：提交与补偿回滚

提交分为全量 staging 与逐个替换两步。`commit_outputs` 先拒绝重复目标，然后在每个目标的同一目录中以 `create_new` 创建唯一临时文件，写入完整内容、`sync_all`，并在目标已存在时复制其权限。所有输出都完成 staging 后才会移动任何任务书文件；任一临时文件创建或准备失败会清理本次临时文件，并在任务书未修改的状态下终止。

替换已有目标时，先把旧目标 rename 到唯一的同目录 rollback 文件，再把临时文件 rename 为目标；新目标则直接把临时文件 rename 到目标。这样不依赖 Windows 的 rename 覆盖已有目标语义。若第 `i` 个替换失败：

- 恢复当前已移走的旧目标，并对此前已提交目标按逆序补偿；
- 原有文件删除新目标后把 rollback 文件 rename 回原路径；
- 原先不存在的目标文件删除；
- 日志记录失败路径、此前已写文件数和回滚错误数；
- 返回的结构化错误明确区分“已恢复”或列出恢复失败，并在任一补偿失败时设置 `task_book_modified=true`。

全部替换成功后才删除 rollback 文件；清理旧 rollback 失败只记录提交后告警，不把已完成写回误报为失败。这提供了可测试、兼容 Windows rename 语义的补偿事务，但跨多个文件仍不具备文件系统原子性。进程若在 `target → rollback → staged target` 两次 rename 之间或多文件替换途中崩溃，仍可能留下 rollback 文件或部分新结果；磁盘/权限持续故障也可能使补偿失败。此时完整备份是人工恢复依据。

## 阶段 7：提交后的附属持久化

任务书全部写入成功后，应用才更新缓存、插入 SQLite 历史并保存 `report-latest.json`。这些操作失败时分别记录 `*_failed_after_commit` 告警；写回仍返回成功，历史编号失败时为 `0`。

这样划分是因为再次回滚已提交的任务书只为弥补非关键报告失败，会扩大数据风险。最终 `cmp_applied` 只在任务书提交完成后记录。强类型响应成功解析后，状态库再把任务从 `applying` 持久化为 `applied`，随后才向界面返回成功；即使界面丢失回包或应用重启，第二次写回仍会被拒绝。若任务书已经提交但状态保存失败，任务保留为 `applying` 并返回 `task_book_modified=true` 的结构化错误，优先阻止重复写回。

应用失败且确认任务书未修改时，状态恢复为 `review_ready`，允许修正后重试；若错误表示任务书可能已修改，则进入 `failed`，不自动开放再次应用。首次导入且状态库中不存在的合法 CMP 会注册为 `review_ready`，但导入不会覆盖已有 `applied` 记录。

## 故障矩阵

| 故障点 | 是否写任务书 | 是否创建备份 | 处理 |
|---|---:|---:|---|
| CMP 解析/身份/源指纹失败 | 否 | 否 | 拒绝 |
| 单元或完整条目格式守卫失败 | 否 | 否 | 拒绝 |
| 最终 SNBT/章节结构无效 | 否 | 否 | 拒绝 |
| 备份父目录/快照创建失败 | 否 | 否 | 拒绝 |
| 备份复制失败 | 否 | 失败快照会清理 | 拒绝 |
| 临时文件创建/准备失败 | 否 | 是 | 清理 staging 后拒绝 |
| 第 N 个目标替换失败 | 已尝试替换前 N 个 | 是 | 恢复当前目标并逆序补偿；补偿失败标记可能已修改 |
| 缓存、历史或报告保存失败 | 是，且视为成功 | 是 | 告警，不回滚 |

## 相关测试

`core.rs` 的确定性故障注入覆盖：备份父目录/快照/复制失败、临时文件创建和准备失败、第二文件替换失败、已有目标最终替换失败、补偿失败、成功后无临时产物，以及历史/报告失败仍返回成功。Golden fixture 另以 `commit_failure_restores_earlier_file_from_fixture` 验证多文件逆序恢复；`cmp_with_modified_english_never_writes_output` 和 `invalid_late_record_in_multi_file_cmp_writes_nothing` 验证受保护内容或后置记录失败时零写入。更多验证层次见 [测试策略](testing-strategy.md)。
