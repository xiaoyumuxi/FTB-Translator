# CMP 写回事务

写回从用户确认应用 CMP 开始。这里的“事务”指应用层的预验证、备份、内存生成、提交快照和失败补偿；它不是由文件系统提供的跨文件原子事务。

## 安全目标

- API 请求和人工校对期间不修改任务书；
- 任何身份、源内容、位置或格式校验失败时，不创建备份、不写文件；
- 所有目标文件都能生成合法结果后才进入提交；
- 多文件提交中途失败时恢复本次提交前的内容；
- 已成功提交任务书后，历史或报告保存失败不能把成功误报为写回失败。

## 阶段 1：加载与身份校验

`cmp-apply` 先解析 CMP v1，再解析用户当前选择的任务书目录。应用要求 CMP 的规范化 `quests_dir` 和当前目录一致、`mode` 一致。早期 v1 没有 `task_id` 时补一个操作编号，否则沿用 CMP 编号。

此阶段仅读取文件。CMP 头、meta、状态、JSON 位置行、翻译对语法或重复 `(entry_id, path)` 无效时立即拒绝。

## 阶段 2：重新扫描源文件

应用不会相信生成 CMP 时留在内存里的扫描结果。它重新读取 `lang/en_us.snbt` 或排序后的 `chapters/*.snbt`，用与翻译阶段相同的 `prepare_entry` 规则重建条目和富文本单元，然后比较：

1. `total_entries`；
2. 由条目 ID、NUL 分隔符、完整英文和 `0xff` 分隔符计算的 SHA-256 `source_fingerprint`；
3. CMP 记录数与当前翻译单元数；
4. 每个 `(entry_id, path)` 是否存在且唯一；
5. `file` 是否由当前模式和条目 ID 推导得到；
6. 箭头左侧英文是否与当前单元原文完全相同。

目录、模式、任务书内容、文件归属、条目位置或英文被改动时，流程在任何备份之前终止。

## 阶段 3：译文与完整条目校验

每条右侧译文必须非空，并再次经过换行、格式 token 和富文本结构守卫。然后应用按 JSON Pointer 重建完整富文本条目，再对完整英文/译文运行同一守卫。记录状态不是豁免条件。

非 `translated` 状态仍保持英文时允许应用，但会进入最终报告的人工检查告警；这与“格式不安全仍强行写入”不同，英文原文本身仍须通过所有守卫。

## 阶段 4：在内存中生成全部输出

`lang` 模式复制当前有序 map，替换文本或按换行拆回字符串数组，生成完整 `zh_cn.snbt` 字符串，再用 `snbt::parse` 重新解析。目标只有 `lang/zh_cn.snbt`，源 `en_us.snbt` 不修改。

`chapters` 模式按源文件聚合 `(segment.index, target)`，要求位置无重复、全部存在；按字节 span 从后向前替换并正确转义原引号。每个章节在替换前后检查字符串和 `{}[]()` 结构。直到所有 `FileOutput` 都构造成功，磁盘仍未发生写回。

## 阶段 5：备份

所有验证通过后，应用才在 `.ftb-translater/backups/` 下原子争用一个未使用的目录名。基础名为本地时间 `YYYYMMDD-HHMMSS`，同秒冲突时追加 `-001` 至 `-999`，避免覆盖已有备份。

`lang` 模式复制整个当前 `lang/`，`chapters` 模式复制整个当前 `chapters/`。任一复制失败会终止提交；已创建但不完整的备份目录当前不会自动清理，应保留错误供人工检查。

## 阶段 6：提交与补偿回滚

提交前，`commit_outputs` 为每个目标读取快照：已有文件保存原始字节，不存在的文件记录为 `None`。任何快照读取失败都会在写入前终止。

随后按 `pending_outputs` 顺序写文件。若第 `i` 个写入失败：

- 对 `0..=i` 的快照按逆序恢复；
- 原有文件写回原始字节；
- 原先不存在的目标文件删除；
- 日志记录失败路径、此前已写文件数和回滚错误数；
- 返回的错误明确区分“已恢复”或列出恢复失败。

这提供了可测试的补偿事务，但仍有边界：`fs::write` 本身不是临时文件 + rename；进程在写入或回滚期间崩溃、磁盘/权限持续故障时无法保证原子性。此时完整备份是人工恢复依据。

## 阶段 7：提交后的附属持久化

任务书全部写入成功后，应用才更新缓存、插入 SQLite 历史并保存 `report-latest.json`。这些操作失败时分别记录 `*_failed_after_commit` 告警；写回仍返回成功，历史编号失败时为 `0`。

这样划分是因为再次回滚已提交的任务书只为弥补非关键报告失败，会扩大数据风险。最终 `cmp_applied` 只在任务书提交完成后记录。

## 故障矩阵

| 故障点 | 是否写任务书 | 是否创建备份 | 处理 |
|---|---:|---:|---|
| CMP 解析/身份/源指纹失败 | 否 | 否 | 拒绝 |
| 单元或完整条目格式守卫失败 | 否 | 否 | 拒绝 |
| 最终 SNBT/章节结构无效 | 否 | 否 | 拒绝 |
| 备份创建或复制失败 | 否 | 可能有不完整目录 | 拒绝 |
| 输出快照读取失败 | 否 | 是 | 拒绝 |
| 第 N 个输出写入失败 | 已尝试写前 N 个 | 是 | 逆序补偿回滚 |
| 缓存、历史或报告保存失败 | 是，且视为成功 | 是 | 告警，不回滚 |

## 相关测试

当前 `core.rs` 测试覆盖修改英文时零写入、CMP 校验后生成语言文件、多文件后续写入失败时恢复先前文件、同秒备份不覆盖，以及提交后历史失败不误报写回失败。更多验证层次见 [测试策略](testing-strategy.md)。
