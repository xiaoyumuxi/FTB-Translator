# ADR-005：限流项重试前重新验证任务书一致性

- 状态：接受并已实施
- 决策日期：2026-07-14，2026-07-15 加固
- 历史依据：`c1fd66b`、`5ecd891`

## 背景

提供商在最多三次 HTTP 尝试后仍可能返回 429 或 rate limit。CMP 把这些单元标为 `rate_limited`，用户可以稍后只重试它们，同时保留其他机器译文和人工修改。重试可能发生在文件被编辑、应用重启或用户扫描了另一个整合包之后。

## 原始问题

CMP 的 `entry_id` 和 JSON Pointer 只有在它对应的源任务书版本中才有意义。若直接按旧 CMP 的限流记录重新请求，当前任务书可能已经：

- 换成另一个目录或模式；
- 增删、重排或修改英文条目；
- 改变 JSON 富文本结构和 Pointer；
- 让相同 ID 指向不同源文本。

此时新译文写回旧记录会制造“看似成功但位置身份错误”的 CMP，并可能在之后的应用阶段造成混淆。只在最终写回时拒绝太晚，因为重试已经产生费用、覆盖 CMP 并可能丢失人工上下文。

## 考虑过的方案

1. 不提供选择性重试，要求整个任务重新翻译。简单但浪费请求并覆盖校对工作。
2. 直接信任 CMP 中的英文和位置，不读取当前任务书。快速但存在陈旧身份风险。
3. 允许重试先更新 CMP，最终应用时再验证。能保护任务书，却不能避免错误请求和 CMP 污染。
4. 在发出任何重试请求前，重新扫描当前任务书并执行与应用阶段一致的源身份验证；只更新验证通过的 `rate_limited` 位置。

## 最终选择

采用方案 4。重试前验证规范化目录、模式、条目总数、源 SHA-256 指纹、记录数、`(entry_id, path)`、文件归属和英文原文。没有限流项也拒绝启动。验证通过后待请求集合仅保留 `rate_limited` 位置；合并结果时仅替换这些记录，其他译文和人工编辑保持不变。

## 选择原因

- 在产生网络副作用和修改 CMP 之前失败，符合 fail closed 原则。
- 复用当前解析/拆分规则，确保请求的源文本与未来写回位置来自同一次扫描。
- 保留非限流结果和人工校对，减少成本与重复劳动。
- 源指纹提供整包快速一致性约束，逐记录检查又能给文件归属、位置和英文更具体的保护。

## 缺点

- 每次重试都要重新读取和解析整个任务书，大包会增加少量启动时间。
- 任何源文本变化都会拒绝，即使变化与限流项无关；用户需要重新翻译或重新建立 CMP。
- `source_fingerprint` 依赖当前条目 ID、顺序和完整英文，解析规则变化也可能令旧 CMP 不再可重试。
- 429 识别当前依赖错误字符串包含 `HTTP 429` 或 `rate limit`，不同提供商错误格式可能被归为 `request_failed`。

## 后续影响

- provider 重试和 UI 重试必须区分：前者是单次批次内部最多三次 HTTP 尝试，后者是 CMP 生命周期中的选择性再次执行。
- 重试验证应与 CMP 应用的源验证保持同步；新增身份字段时两处都要更新测试。
- 测试必须覆盖另一个任务书、修改英文、变化 Pointer/文件归属、无 `rate_limited` 记录，以及非目标记录/人工译文不变。
- 重试仍只更新 CMP，不备份、不写任务书；用户之后应用 CMP 时还要再验证一次，因为重试与写回之间仍可能发生变化。
