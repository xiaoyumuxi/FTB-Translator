# ADR-004：以 CMP 作为翻译与写回之间的中间格式

- 状态：接受并已实施（CMP v1）
- 决策日期：2026-07-14
- 历史依据：`f7b9e05`、`1f31699`、`c1fd66b`、`5ecd891`

## 背景

早期流程在调用接口后直接准备任务书输出，人工修正主要发生在结果页。当前产品需要让用户先查看、导出、用文本编辑器或应用表格校对机器译文，并允许应用重启后继续，而翻译阶段不能触碰源文件。

项目已经用 SQLite 保存已完成任务的历史，但历史数据库面向应用内部查询和导出，不是便携、可审查的翻译工程文件。

## 原始问题

如果接口结果直接写回，外部服务错误、格式守卫漏检和术语问题会立刻进入任务书；用户也无法在写回前审阅。若只把待审内容放在内存或数据库，离线编辑、版本控制、另存和跨重启继续会变得不透明，而且很难规定“哪些字段允许人工修改”。

## 考虑过的方案

1. 翻译成功后直接备份并写回，再通过历史恢复或二次修正。
2. 把待审译文全部存入现有 SQLite，并只提供应用内编辑。
3. 使用通用 CSV/JSON。易处理，但人手编辑时不够清楚哪些字段受保护，JSON 整体结构也容易被无意改动。
4. 定义面向人工校对的文本 CMP：受保护的头/meta/位置/英文，加上唯一可编辑的 JSON 字符串译文。

## 最终选择

采用 CMP v1 作为 API 阶段与应用阶段之间的唯一用户可见中间格式。核心对照语法保持：

```text
@ {"file":"chapters/a.snbt","entry_id":"a.snbt:0:title","path":"$","status":"translated"}
"English" -> "中文"
```

只允许修改箭头右侧 JSON 字符串。`# meta` 保存非敏感任务身份、源指纹与提供商信息；`@` 保存唯一回填位置和状态。SQLite 继续保存已经应用的运行历史，两者职责不同，不互相替代。

## 选择原因

- 文本格式便于人工阅读、diff、另存、导入和使用普通编辑器。
- JSON 字符串转义可靠表达换行、引号、反斜杠和正文中的 ` -> `。
- 文件/条目/Pointer/英文都显式记录，可在写回前与重新扫描结果逐项核对。
- 机器翻译与文件修改被强制分成两个阶段，“直接覆盖”也不能绕过同一应用路径。
- 比 SQLite 更适合作为可携带校对工件，同时避免把完整应用数据库暴露给编辑者。

## 缺点

- 自定义 parser、serializer、版本和兼容规则需要长期维护。
- 大型任务书的 CMP 较长，文本编辑器可能不如数据库表格高效。
- 人工仍可能破坏受保护行；应用必须给出明确拒绝而不能尝试猜测修复。
- CMP 包含完整待翻译英文和中文，虽不含凭证，仍可能包含整合包文本；用户应按普通项目文件管理。
- v1 记录顺序和分组承载可读性，未来结构变化需版本化而不能静默改变语义。

## 后续影响

- API 阶段只允许生成 CMP、缓存和诊断数据；备份和任务书写回必须在应用阶段。
- CMP parser 必须拒绝未知版本/状态、重复位置和破损对照，并兼容早期 v1 缺少 `task_id`。
- 每次格式变化必须同步 `cmp.rs` 测试、[CMP v1 规范](../cmp-format.md) 和 README。
- CMP 不得包含 API Key、Authorization、钥匙串值或完整 HTTP 请求/响应。
- 导入、表格编辑、限流重试和“直接覆盖”最终都必须重新验证当前任务书。
