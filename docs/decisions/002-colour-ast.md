# ADR-002：颜色码需要轻量样式 AST，而不只是数量比较

- 状态：接受；历史 Python 已实现，当前 Rust 尚未移植
- 决策日期：2026-07-01
- 历史依据：`6b433f7`、`e7fe6a4`、`ffa6bc4`

## 背景

Minecraft 旧式颜色/格式码以 `&` 或 `§` 开头。颜色码会替换当前颜色，`k-o` 等格式码叠加样式，`r` 重置状态。中文语序变化允许完整的带样式短语移动，因此要求所有颜色码保持原绝对位置或原始顺序会产生误报；但只比较数量也不能证明样式语义相同。

历史 Python `format_guard.py` 在 token 多重集合之后增加轻量颜色 AST：扫描文本，维护活动颜色/样式，记录每个有文本 span 的活动样式签名，并检测“无活动样式的 reset”和“末尾悬空样式码”等异常。Rust 重写没有移植该层；当前 `core::warnings` 只比较排序后的受保护 token。

## 原始问题

以下译文可能拥有与原文完全相同的颜色码数量，甚至相同的 token 多重集合，却仍不安全：

```text
原文：&c&lWarning&r text
错误：&l&c警告&r 文本
```

`&c` 会重置活动颜色相关状态，`&l&c` 与 `&c&l` 的作用状态不同。又如把 `&r` 放到实际文本之前、把颜色码留在字符串末尾，计数都可能相同，但样式作用域已经改变或根本没有被应用。

## 考虑过的方案

1. 要求颜色码字节序列和位置完全相同。安全但不允许中文语序导致的合理短语移动。
2. 只比较颜色码数量或排序后的多重集合。宽松，但无法识别作用域和顺序语义。
3. 把颜色码交给模型修复或自动增删。不可确定，且可能掩盖新的格式损坏。
4. 建立只描述活动样式 span 的轻量 AST，再比较样式签名和新增语法异常。

## 最终选择

采用方案 4 作为颜色码的第二层守卫：token 保护/恢复保证没有增删，轻量 AST 判断相同 token 是否形成等价的活动样式 span。完整 JSON 富文本仍使用 JSON 结构树和 Pointer 回填，不与旧式颜色 AST 混为一个通用 AST。

这是历史上已接受并实现过的决策，但当前 Rust 版本只保留了第一层多重集合检查。当前产品不能声称已执行颜色 AST 等价验证。

## 选择原因

轻量 AST 比绝对位置比较更适应翻译语序变化，又比单纯计数能表达颜色替换、格式叠加、reset 和悬空 token。它只解析很小的 Minecraft 样式状态机，不要求引入完整渲染器。

## 缺点

- 必须准确维护每种 legacy code 的状态语义；模组自定义码（例如历史实现包含的 `z`）可能与原版不同。
- 比较“样式签名的多重集合”仍不理解每段文字的语义重点，可能允许样式从一个词移动到另一个词。
- JSON 组件中的继承样式比平面字符串复杂，需要与结构守卫明确分层。
- 当前 Rust 缺少实现和回归测试，存在已知保护缺口。

## 后续影响

- Rust 端应移植小型状态机，并加入“token 相同但顺序/作用域不同”的拒绝测试。
- 在实现前，README 的格式守卫通过率不能解释为颜色样式安全率，更不能解释为语义准确率。
- 自动修复最多应删除可证明为多余的 token，不能凭空补码；当前 Rust 实际采用整条回退英文，不做样式自动修复。
- 对 JSON 富文本继续比较非展示结构；颜色、样式字段保持原值，只有允许的展示字符串可替换。
