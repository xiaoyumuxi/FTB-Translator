# 翻译流水线

本文描述当前从扫描到生成 CMP、以及限流项重试的后端数据流。CMP 应用和磁盘提交见 [写回事务](writeback-transaction.md)。

## 阶段总览

```text
定位任务书 → 选择模式 → 提取条目 → 拆分富文本展示单元
    → token/词表保护 → 缓存分流 → 分批并发请求
    → 占位符恢复 → 重建条目 → 格式守卫 → 写缓存与 CMP
```

流水线以 `task_id` 串联前端事件、提供商批次、CMP、写回和历史日志。普通翻译创建新编号；重试已有 CMP 时复用其非空 `task_id`。早期缺少编号的 v1 文件根据规范化任务书路径、模式、源指纹和受保护条目身份生成跨重启稳定的内部编号；人工修改右侧译文不会改变该编号。

## 1. 定位、模式和提取

`core::resolve` 接受整合包、`config`、`ftbquests`、`quests`、`lang` 或 `chapters` 等常见选择位置，规范化为包含任务书的目录。搜索深度当前最多为五层。`lang/en_us.snbt` 存在时选择 `lang`；否则选择非空的 `chapters/*.snbt`。

`lang` 模式由手写 SNBT 子集解析器读取。根值只支持字符串与字符串数组；数组在内部合并为带真实换行的一个条目，应用时再按换行拆回数组。

`chapters` 模式按文件名排序。当前实现用受限字段正则提取字符串字面量，忽略当前规则识别到的注释，并仅保留含 ASCII 英文字母的值。每个片段记录原文件、字段名、字节 `start/end`、原引号、文件内序号和稳定的 `cache_id`。这是面向已知 FTB 字段的提取器，不是完整 SNBT 语法解析。

## 2. 富文本拆分

每个条目进入 `prepare_entry`：

- 普通字符串成为一个 `path="$"` 单元；
- 可解析的 JSON object/array 由 `rich_text::Document` 遍历，只提取玩家可见的根字符串、`text`、`extra`、`with`、`separator` 和允许的 `hoverEvent` 展示内容；
- `clickEvent`、`translate`、`keybind`、样式、颜色、命令、URL 和资源 ID 不作为可翻译字段；
- JSON object 出现重复键时拒绝解析；看起来像组件但不能安全解析时整条保留原文，不降级成普通文本交给接口。

每个富文本单元保存 JSON Pointer。重建时只在这些 Pointer 上替换字符串，随后以“把所有展示字段归一成 `$display_text`”的结构快照比较源与目标。

## 3. 格式和词表保护

`core::protect` 收集颜色/格式码、printf 占位符、标签、宏、选择器、资源 ID、路径、转义序列、URL、十六进制颜色、版本、数字和单位的非重叠 span，按原位置替换为 `⟨P_N⟩`。恢复前会比较预期与实际占位符多重集合；缺失、增加、重复或未知 `P/G` 占位符都会令该单元回退英文。

API 模式可在格式保护之后执行本地词表最长优先匹配，生成 `⟨G_N⟩`；网页模式强制关闭词表。词表是本地保护层，不是提供商官方 glossary。

这里的正则只识别字符串内部的不可翻译 token，不负责解析整个 SNBT 文件。颜色码的当前校验边界见 [ADR-002](decisions/002-colour-ast.md)。

## 4. 缓存和执行计划

缓存以完整条目为单位，而接口可能处理一个条目的多个富文本单元。命中完整条目缓存时跳过它的所有单元；未命中的单元按批大小分组。

- API 模式：`auto` 批大小 25、并发 6；并发最大 12。
- Google/DeepL 网页模式：配置最终规范化为 `auto`，核心层再把有效并发限制为 1。
- Google 网页按约 4,500 字符组包并用不可翻译标记拆回；DeepL 网页按约 1,500 字符装入文本数组。
- 超长文本在标点或空白附近拆分，并避免在 `⟨…⟩` 占位符中间切开。

批次通过 `buffer_unordered` 并发执行，所以完成顺序不保证与输入相同；结果始终用单元 ID 重新关联，CMP 记录仍按原始 `entries/items` 顺序生成。

## 5. 提供商失败与恢复

提供商层对请求最多尝试三次并递增等待。OpenAI 兼容接口还会在对方不支持 `response_format` 时去掉该字段继续尝试。核心层把最终错误含 `HTTP 429` 或 `rate limit` 的批次标为 `rate_limited`，其他请求失败标为 `request_failed`；整批单元回退英文。

请求成功后，核心层恢复 `P/G` 占位符。恢复失败标为 `format_guard` 并回退该单元。富文本单元全部处理后按 Pointer 重建完整条目；重建失败也保留英文。

最后的条目级格式守卫比较：

- 实际换行、回车、制表符数量；
- 重新提取并排序后的受保护 token 多重集合；
- JSON 富文本的键、类型和非展示字段结构。

有告警的完整条目回退英文，只有无告警且发生变化的译文进入缓存。

## 6. 生成 CMP 与诊断文件

API 阶段写入：

- `.ftb-translater/translation-units-latest.jsonl`：本次实际请求单元的诊断结果；
- `.ftb-translater/cache.json`：已通过条目级格式守卫的缓存；
- `.ftb-translater/reviews/translation-<时间>.cmp`：人工校对文件。

CMP 元数据包含任务目录、模式、源指纹、提供商、接口、模型、翻译要求、词表指纹、条目数和缓存命中数，不包含凭证。每条记录保存源文件、条目 ID、JSON Pointer、状态、英文和译文。生成成功以 `review_ready` 结束；此时没有备份，也没有任务书写回。

翻译命令在异步任务启动前，先在系统应用数据目录的状态库中把任务原子转换为 `translating`。同一任务书已有 `translating` 或 `applying` 时拒绝重复启动；CMP 成功落盘后转为 `review_ready`，失败转为 `failed`。限流重试复用同一身份，只允许从 `review_ready` 或 `failed` 进入 `translating`。

## 7. 限流项重试

重试不是“读取 CMP 后直接重新发请求”。开始前重新扫描当前目录并重新构造条目和翻译单元，然后验证：

- CMP 记录的目录与规范化后的当前目录相同；
- 模式相同；
- 条目数与源指纹相同；
- CMP 记录数、回填位置、文件归属和英文原文与当前提取结果一致。

验证通过后只保留状态为 `rate_limited` 的 `(entry_id, path)` 进入待请求集合。新结果只替换这些位置，其他机器译文和人工编辑按原记录保留，并原地更新同一 CMP。设计理由见 [ADR-005](decisions/005-rate-limit-retry-consistency.md)。

## 状态不等于写回权限

`translated`、`unchanged`、`rate_limited` 等状态只服务于筛选和提示。任何状态都不能跳过应用阶段的重新扫描和格式守卫。即使用户选择“直接覆盖”，也必须走与人工校对相同的 CMP 应用命令。
