# FTB Translator 诊断日志规范

本文档面向维护者，记录桌面应用日志的文件布局、级别、事件覆盖、安全边界和扩展方式。README 不承载这些实现细节。

## 文件与位置

日志固定写入应用程序所在目录的 `logs/`，不使用系统用户数据目录：

- Windows/Linux：可执行文件旁的 `logs/`；
- macOS：`.app` 文件旁的 `logs/`。

前后端必须分流：

```text
logs/
├── frontend.log
├── frontend.log.1 ... frontend.log.5
├── backend.log
└── backend.log.1 ... backend.log.5
```

`frontend.log` 接收 React 界面事件，`backend.log` 接收 Rust/Tauri 事件。两份活动日志分别达到 5 MB 后滚动，各自保留 5 份旧文件。设置页的“导出前后端日志”会把现有的两组文件放入同一个 ZIP。

应用目录不可写时，日志系统不得静默改写到 `AppData`、`Application Support` 或 `~/.local/share`。应用可以继续启动，但设置页必须显示初始化错误，打开目录和导出操作也必须返回同一错误。

## 日志级别

前后端共用用户保存的 `log_level`，有效值如下：

| 级别 | 用途 |
|---|---|
| `error` | 无法继续的界面异常、任务失败、未处理异常 |
| `warn` | 可恢复失败、无效操作、HTTP 失败与重试 |
| `info` | 默认级别；任务、设置、凭证、历史和用户操作的关键阶段 |
| `debug` | 页面切换、缓存命中、批次和单次请求尝试 |
| `trace` | 高频进度、命令入口和逐翻译单元信息 |

默认级别为 `info`。改变默认值不能覆盖已有用户在 `settings.json` 中保存的有效级别；无效值加载时回退到默认值，保存时则明确拒绝。

## 日志格式

每条记录占一行，格式为：

```text
2026-07-14T12:34:56.789Z INFO  [translation] task_started - 翻译任务已开始 | {"task_id":"...","provider":"google_web"}
```

字段含义：

- UTC 毫秒时间戳；
- 大写日志级别；
- 后端模块名或固定的 `frontend`；
- 稳定的英文事件名；
- 简短中文说明；
- JSON 结构化上下文。

事件名只允许 ASCII 字母、数字、点、连字符和下划线，最长 80 个字符。消息会被压成单行并截断；结构化对象限制递归深度、数组长度和字符串长度。

## 前端事件覆盖

前端通过 `frontend-log` Tauri bridge 写入 `frontend.log`。目前覆盖：

- React 启动、渲染错误边界；
- `window.error` 和未处理的 Promise rejection；
- 前端调用后端命令失败；
- 主题与页面切换；
- 目录选择、扫描开始和扫描结果；
- 翻译启动、逐条进度、限流批次重试、CMP 校对就绪、打开、另存、重新选择、人工校对选择、最终写回完成和失败事件；
- 提供商切换和设置保存；
- 词表选择与默认路径恢复；
- 用户明确查看凭证的状态，不记录凭证内容；
- 历史删除、历史导出和写回后的人工修正；
- 打开日志目录与导出诊断 ZIP。

前端不得把完整设置对象写入日志，因为其中可能包含当前会话的 API Key。调用失败只记录命令名和清洗后的错误，不记录 bridge payload。

## 后端事件覆盖

后端直接写入 `backend.log`。目前覆盖：

- 应用与日志系统启动；
- Tauri bridge 命令入口和失败；
- 普通设置加载与保存；
- 系统凭证的按需读取、会话缓存命中、保存和删除；
- 翻译历史的读取、写入、删除和导出；
- 任务书扫描；
- 翻译任务开始、内容解析、执行计划与 CMP 校对文件生成；
- 带 `task_id` 的批次开始、完成、失败与耗时；
- 提供商 HTTP 尝试、状态失败、网络失败和重试；
- CMP 解析、应用开始、安全校验通过或失败；
- 用户确认应用 CMP 后的备份开始与完成、输出提交开始与完成、提交失败与回滚结果，以及最终历史编号；
- 后端任务状态的原子转换与 CMP 状态加载；只记录非敏感任务身份和状态名，不记录 CMP 正文；
- 输出已经提交后，缓存、历史或最新报告保存失败的非破坏性警告；这些警告不得被记录成任务书写回失败。

翻译日志必须区分两个阶段：

1. API 阶段以 `review_ready` 结束，只能记录 CMP 路径、条目/缓存/失败/告警数量和耗时，不得记录“输出已提交”或“任务书已覆盖”。
2. 应用阶段以 `cmp_applied` 结束，只有这个阶段可以记录备份、输出文件数和历史 `run_id`。

“是，直接覆盖”和“否，人工校对”最终使用相同的 CMP 应用命令，因此后端安全校验和提交事件必须保持一致。CMP 元数据保存 API 阶段生成的 `task_id`，应用重启后导入仍使用该编号串联；兼容缺少该字段的早期 v1 文件时，使用规范化任务书身份、源指纹和受保护条目身份生成稳定内部编号。打开、导出或重新选择 CMP 时只能记录路径和结果，不得记录 CMP 的英文或中文正文。

同一次翻译必须复用 `task_id`，使任务级事件和批次事件可以串联。`trace` 可以记录条目 ID 与 JSON Pointer，但不得记录原文或译文内容。

## 敏感信息边界

任何级别都禁止记录：

- API Key、Authorization、密码、Token、Secret；
- 完整 HTTP 请求体或响应体；
- 完整翻译提示词；
- 完整任务书原文或译文；
- 钥匙串返回的凭证值。

后端会递归遮盖常见敏感字段，包括下划线、连字符和驼峰形式的 API Key、Authorization、密码、访问/刷新 Token 与客户端 Secret。名为 `error` 的 HTTP 错误只保留状态摘要，避免第三方响应体回显待翻译文本。调用方仍必须只传入必要的诊断元数据，不能依赖脱敏器兜底不当日志。

## 扩展规则

新增功能时按以下方式补充日志：

1. 用户在界面触发的关键动作，在前端记录开始、取消或可见结果。
2. 对应的 Tauri 命令、持久化或外部请求，在后端记录执行结果。
3. 高频循环只使用 `debug` 或 `trace`，避免默认 `info` 产生大量日志。
4. 错误事件记录稳定的事件名与错误类别，不记录请求/响应正文。
5. 涉及翻译任务的事件携带 `task_id`；涉及历史记录的事件携带 `run_id`。
6. 更新文件名、滚动策略、级别或脱敏规则时，同步修改本文档和相关单元测试。

最低验证：

```bash
cargo fmt --manifest-path src-tauri/Cargo.toml -- --check
cargo test --manifest-path src-tauri/Cargo.toml
npm run build
git diff --check
```

日志测试至少要验证：五种级别解析、敏感字段遮盖、滚动顺序，以及导出的 ZIP 同时包含 `frontend.log` 和 `backend.log`。
